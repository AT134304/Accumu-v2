# Accumu — 프로젝트 메모리 (v2: 실제 작동 프로토타입)

이 파일은 Claude Code가 세션을 시작할 때마다 자동으로 읽습니다. 모든 서브에이전트(`.claude/agents/*.md`)도 이 내용을 함께 로드하므로, 원칙이나 구조가 바뀌면 반드시 여기부터 수정하세요.

> 이 문서는 `Accumu_기획서_v2.docx`(학생 화면·관리자 화면 통합 기획서)를 코드 작업용으로 요약한 버전입니다. 세부 내용이 다르면 원본 기획서가 우선합니다.

## 1. 프로젝트 성격 (중요)

Accumu는 **사업 출시가 아니라 입시 포트폴리오용으로 직접 시연 가능한 "실제로 작동하는" 프로토타입**입니다. 기존 정적 HTML 데모(새로고침하면 데이터 초기화)에서 한 단계 나아가, 실제 로그인·데이터 저장·QR 카메라 인증이 동작하는 수준까지 구현합니다. 제작자 1인이 학생 계정과 관리자 계정을 오가며 시연하는 것이 목표이며, 대규모 사용자·수익모델·법적 동의 절차 등은 스코프 밖입니다 (10장 "스코프 제외 사항" 참고).

## 2. 절대 원칙 (모든 에이전트가 매 작업마다 지켜야 함)

1. **게임적 요소·사행적 요소 완전 배제.** 랜덤박스, 뽑기, 경쟁 랭킹, 아바타 성장, 베팅 없음. 별점 리뷰, 담백한 마일스톤 표시 정도만 허용.
2. **학업(자습·문제풀이) 아니라 진로·커리어 활동에 집중.**
3. **포인트→지역화폐 전환은 시뮬레이션만.** 실제 결제·계좌 연동 없음. 전환 시 포인트 잔액 차감 + 지역화폐 누적액 표시까지만 구현.
4. **포트폴리오가 포인트보다 먼저 보여야 함.** 새 화면을 만들 때 포인트(amber)보다 성장/포트폴리오(brand blue) 요소를 시각적으로 우선.
5. **QR은 입장·퇴장 2회 인증.** 부정 참여 방지 장치이므로 단순화하지 않음. GPS 인증은 이번 스코프에서 제외.
6. **관리자 기능은 아래 3가지로 한정.** 프로그램 올리기/내리기/수정, 담당 학생 5명 아카이브 조회(PDF 확인 포함), QR 카메라 스캔. 학교 단위 대시보드나 다중 관리자 관리는 넣지 않음.

## 3. 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| 프런트엔드 | React (Vite) | 기존 vanilla HTML 프로토타입(`Accumu_prototype.html`)의 화면·인터랙션을 참고해 재구현 |
| 백엔드 | Supabase (PostgreSQL + Auth) | 별도 서버 없음. RLS로 권한 제어 |
| QR 생성 | qrcode.react | 학생 측 QR 표시 |
| QR 인식 | html5-qrcode (또는 동급 카메라 스캔 라이브러리) | 관리자 측, 실제 카메라 권한 사용 |

기존 `Accumu_prototype.html`은 삭제하지 않고 화면 로직·카피·디자인 참고용 레퍼런스로 유지합니다.

## 4. 역할(Role) 구조

- **student**: 프로그램 탐색·신청·QR 인증(피스캔)·아카이브 열람·포인트 관리
- **admin**: 프로그램 등록/게시중단·QR 스캔(인증 처리)·담당 학생 아카이브 열람

로그인 화면 상단에 "학생 / 관리자" 탭이 있고, 탭에 따라 입력 필드와 인증 방식이 다릅니다.

| 구분 | 입력 필드 | Auth 처리 |
|---|---|---|
| 학생 | 학번·이름·비밀번호 | `{학번}@accumu.local` 가상 이메일로 변환 |
| 관리자 | 관리자 코드(예: `ADM-0001`)·비밀번호 | `{코드}@accumu.local` 가상 이메일로 변환 |

`profiles.role`(student/admin)로 화면을 분기하고, 프로그램 등록·게시중단·QR 스캔 처리는 Supabase RLS로 `role = admin`만 허용합니다.

## 5. 데이터 모델

| 테이블 | 주요 필드 | 설명 |
|---|---|---|
| `profiles` | id, role, code, name, points_balance, points_total, currency_balance, career_interest | 학생/관리자 공통 계정 |
| `mentor_students` | admin_id, student_id | 관리자-담당학생 고정 매핑 (데모 기준 5명) |
| `programs` | id, category, title, description, date, time, capacity, points, is_published, created_by | is_published로 게시/게시중단 |
| `participations` | id, student_id, program_id, status, entry_at, exit_at, entry_token, exit_token | 신청·입장·퇴장 상태 + QR 토큰 |
| `point_transactions` | id, student_id, type(적립/전환), amount, related_participation_id | 포인트 내역 |
| `reviews` | id, participation_id, rating, comment | 별점 + 한줄평 |
| `notifications` | id, student_id, type, message, is_read, created_at | 인앱 알림 |

## 6. QR 이중 인증 작동 방식

1. 학생이 "입장 QR 받기" → `{participation_id, type: entry, expires_at: 발급+30분, 1회용 token}`을 QR로 인코딩해 표시
2. 관리자가 카메라로 스캔 → 토큰 검증(만료 여부·재사용 여부) → 통과 시 `entry_at` 기록, 토큰 즉시 무효화
3. 퇴장도 동일한 방식. 퇴장 인증 완료 시 포인트 자동 지급 + 학생 화면에 만족도 평가(별점+한줄평) 자동 노출
4. 만료/재사용 토큰은 스캔 시 거부하고 사유를 관리자 화면에 표시

## 7. 포인트 규칙

- 최소 150P, 최대 3,000P, 모든 값 끝자리 0
- 분포는 150~700P 구간에 두텁게, 2,000~3,000P는 극소수 활동만
- 1P = 1원 개념이나 실제 결제 연동 없음

## 8. 디자인 시스템

```
--ink: #16213E        --bg: #EAEEF6
--brand: #2563EB      --brand-dk: #1D4ED8      --brand-soft: #DEE9FE
```
초록 계열 금지. 포인트는 amber로 절제, 포트폴리오/성장 요소는 항상 blue 우선. 아이콘은 이모지 금지, duotone SVG. 반응형 브레이크포인트 980/768/600/430px, 모바일은 하단 탭바.

## 9. 학생 화면 (10개, 상세는 기획서 4장)

로그인 → 메인 → 프로그램 선택 → 참여 팝업 → QR 입·퇴장 인증 → 만족도 평가 → 디지털 아카이브 → 마이페이지(포인트/지역화폐) / 상단 공통: 알림, 캘린더(기본값 = 실제 오늘 날짜).

## 10. 관리자 화면 (기획서 5장)

관리자 홈(오늘 진행 프로그램 우선) / 프로그램 관리(올리기·내리기·수정) / 담당 학생 5명 아카이브 조회 + PDF 확인 / QR 카메라 스캔.

## 11. 스코프 제외 사항 (구현하지 않음)

실제 지역화폐 결제·계좌 연동, 법정대리인 동의 화면, GPS 인증, 백그라운드 푸시 알림, 다중 관리자·학교 단위 대시보드, 1365 등 외부 봉사 API 실연동.

## 12. 작업 방식 (에이전트 팀)

실제 백엔드가 생기면서 작업 단계가 늘어나 에이전트를 5개로 구성했습니다 (`.claude/agents/` 참고):

1. **pm-agent** — 새 기능/수정 요청이 들어오면 가장 먼저 호출. 요구사항을 스펙으로 정리, 절대 원칙 위반 체크.
2. **architect-agent** — 데이터 모델·상태 흐름 설계, ADR을 `docs/adr/`에 기록.
3. **backend-agent** — Supabase 스키마(마이그레이션 SQL), RLS 정책, Auth 설정, QR 토큰 발급/검증 로직.
4. **frontend-agent** — React 컴포넌트 구현, 기존 프로토타입 디자인 재사용.
5. **qa-agent** — 구현 후 검증 (원칙 준수, 권한 누수, 반응형, 데이터 정합성).

기본 흐름: `pm-agent`(스펙) → `architect-agent`(설계, 필요시) → `backend-agent`(스키마/API) → `frontend-agent`(화면) → `qa-agent`(검증). 단순 UI 수정은 pm-agent 없이 바로 frontend-agent를 불러도 됩니다.
