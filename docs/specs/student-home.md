# 학생 메인 화면 (홈)

## 영향 범위
학생 화면 (로그인 성공 직후 첫 진입 화면). 관리자 화면 영향 없음.
※ 이번 스펙에서 학생 공통 셸(상단 네비게이션 + 모바일 하단 탭)을 처음 만들지만, 셸이 링크하는 다른 학생 화면(프로그램 선택/아카이브/마이페이지)은 이번 스코프 아님 → placeholder.

## 배경 / 왜 필요한가
로그인은 완성됐지만 `StudentHomePage.jsx`는 이름/역할 표시 + 로그아웃만 있는 확인용 골격임. 로그인 이후 학생이 실제로 보게 되는 첫 화면을 채워야 이후 흐름(프로그램 선택 → 참여 → QR → 아카이브)의 진입점이 생김.
이 화면은 앱의 첫인상이라 **절대 원칙 4번("포트폴리오가 포인트보다 먼저 보여야 함")**을 시각적으로 증명하는 자리다. 기존 프로토타입 `Accumu_prototype.html`의 `renderMain()`(845줄) 화면 구성을 레퍼런스로 재현하되, `programs` 테이블이 아직 없으므로 이번 기능에서 처음 설계·시드한다.

## 사용자 시나리오
1. 학생이 로그인에 성공하면 `/student`(홈)로 진입, 상단에 로고·네비·알림·캘린더·포인트·아바타가 있는 공통 셸과 함께 히어로 영역을 본다.
2. 히어로에서 "참여가 쌓여 나의 커리어가 된다" 카피와 CTA 버튼("프로그램 둘러보기", "내 아카이브 보기")을 보고, 옆(또는 아래)에 지금까지 쌓인 활동을 담백하게 보여주는 마일스톤 스택 시각화를 본다.
3. 아래로 스크롤하면 "이렇게 작동해요" 성격의 개요 카드 3종(활동 참여 / 지역화폐 포인트 / 디지털 포트폴리오)을 본다.
4. 그 아래 "추천 프로그램" 섹션에서 본인의 관심 진로 계열(`profiles.career_interest`)에 맞춰 골라진 프로그램 카드들을 가로로 훑어본다. 계열이 일치하는 카드에는 "내 관심 계열" 배지가 붙는다.
5. "전체 보기 →" 또는 카드 클릭 시 프로그램 선택 화면으로 이동한다(이번 스코프에서는 placeholder 라우팅까지).
6. 모바일에서는 하단 탭바(홈/프로그램/아카이브/마이)로 화면을 전환한다.

## 화면 구성 요소

### A. 상단 공통 셸 (이번 스펙에서 신규 제작, 이후 모든 학생 화면 공용)
프로토타입 `<nav class="nav">`(530~547줄) + `<nav class="bottomnav">`(670~675줄) 기준.
- 좌측: Accumu 로고 (클릭 시 홈)
- 중앙(데스크톱): 네비 메뉴 — 프로그램 / 디지털 아카이브 / 마이페이지
- 우측:
  - 캘린더 아이콘 버튼 (기본값 = **실제 오늘 날짜**. 프로토타입은 `TODAY_ISO='2026-07-02'`로 하드코딩돼 있으나 CLAUDE.md 9장 원칙대로 실제 오늘로 대체)
  - 알림 아이콘 버튼 (안 읽은 알림 표시 dot)
  - 포인트 표시: amber 코인 + `points_balance` 값 + `P` (nav 우측 구석, **작게/절제**)
  - 아바타: 이름 첫 글자 이니셜 (정적. 성장/레벨 요소 아님)
- 모바일: 하단 탭바 4개 (홈/프로그램/아카이브/마이)

### B. 메인 본문 (screen-main, 551~581줄)
1. **히어로** — 카피("참여가 쌓여 / 나의 커리어가 된다"), 설명 문단, CTA 버튼 2개(프로그램 둘러보기 / 내 아카이브 보기), 마일스톤 스택 시각화(`stackviz`). 성장/포트폴리오 서사이므로 brand blue 우선.
2. **개요 카드 3종** — 활동 참여(blue) / 지역화폐 포인트(amber) / 디지털 포트폴리오(indigo). 포인트 카드는 3장 중 1장으로만 등장(포인트 과대노출 방지).
3. **추천 프로그램 섹션** — 섹션 헤더("추천 프로그램" + "OOO님의 관심 분야를 바탕으로 골라봤어요" + "전체 보기 →") + 프로그램 카드 가로 스크롤 행. 카드는 프로토타입 `pcardHTML()`(815줄) 재현: 카테고리 태그·제목·주최·일시·포인트(+NNN P, amber)·참여 버튼·계열 매칭 배지.

## 기능 요구사항 (체크리스트)
- [ ] 로그인 성공 후 `/student`에서 공통 셸 + 메인 본문 렌더 (현재의 확인용 골격을 대체)
- [ ] 상단 셸: 로고, 네비 메뉴, 캘린더/알림 아이콘, 포인트 표시, 아바타 이니셜 배치
- [ ] 캘린더 기본값은 실제 오늘 날짜(`new Date()`) 기준 (하드코딩 금지)
- [ ] 포인트 표시는 로그인 학생의 `profiles.points_balance` 실제 값 사용, amber로 절제 표기
- [ ] 히어로 카피/CTA/마일스톤 스택 시각화 렌더 (brand blue 우선)
- [ ] 개요 카드 3종 렌더 (포인트 카드는 1/3로 제한)
- [ ] 추천 프로그램: `programs`에서 `is_published=true` 프로그램을 조회해 카드로 렌더
- [ ] 추천 정렬: 학생 `career_interest`와 프로그램 계열이 일치하는 것 우선, 이어서 나머지 (계열 정보 미도입 시 최신순 fallback — 아래 열린 질문 E 참고)
- [ ] 계열 일치 카드에 "내 관심 계열" 배지 표시
- [ ] 지난 날짜/미게시 프로그램은 추천에서 제외
- [ ] 추천 결과가 없을 때 빈 상태 문구 노출("지금 추천할 새 프로그램이 없어요")
- [ ] 반응형: 980/768/600/430 브레이크포인트, 모바일 하단 탭바 동작
- [ ] 네비/CTA/카드 클릭의 라우팅 목적지 처리 (아래 스코프 구분 참고)

## 데이터/백엔드 영향 (→ architect-agent 검토 필요)
`programs` 테이블을 이번 기능에서 처음 생성한다. CLAUDE.md 5장 기본 필드 + 프로토타입 재현에 필요한 추가 필드 여부를 architect-agent가 확정해야 함.

**CLAUDE.md 5장 기본 필드**: `id, category, title, description, date, time, capacity, points, is_published, created_by`

**프로토타입 재현에 필요한데 5장에 없는 필드 (architect 결정 필요)**:
| 후보 필드 | 프로토타입 근거 | 메인 화면에서의 용도 | PM 의견 |
|---|---|---|---|
| `org` (주최/기관명) | 모든 카드에 표시 (`p.org`) | 카드마다 주최 표기 | **추가 권장** (카드 필수 정보) |
| `career_track` (진로 계열) | `p.track` (sci/it/hum/biz/art) | 추천 계열 매칭 + "내 관심 계열" 배지 | **추가 권장** (없으면 추천이 최신순으로만 동작) |
| `popularity` 또는 참여자 수 | `p.pop` (인기순 정렬·추천 우선순위) | 추천/인기순 정렬 | 정적 필드 vs `participations` 카운트 집계 중 택1 → architect 결정 |
| `status` (open/ing/wait/full/over) | `p.status` (참여 버튼 상태) | 카드 참여 버튼 라벨/비활성 | 정적 필드 vs (date+capacity+참여수) 파생 중 택1 → architect 결정. 메인 화면은 표시만 필요 |

- 추천 정렬을 위해 `profiles.career_interest`(단일 계열) ↔ `programs.career_track` 매핑 축이 필요. 계열 taxonomy(프로토타입 TRACK: 이공계·자연과학 / IT·SW / 인문·사회 / 경영·경제 / 예술·체육)를 그대로 쓸지 architect가 확정.
- 알림/캘린더 팝업의 실제 데이터(`notifications`, `participations`)는 이번 스코프에서 다루지 않음 → 아이콘/셸만 배치 (열린 질문 C).
- RLS: `programs`는 학생이 `is_published=true` 행을 select 가능해야 함 → backend-agent 위임.
- 시드: 데모 프로그램 시드 데이터 필요 (규모/카테고리는 열린 질문 F).

## 스코프 아님 (Out of scope)
- **프로그램 선택 화면(전체 카테고리 목록·검색·필터·정렬)** — 프로토타입 `screen-programs`(584~602줄)는 별도 화면. 다음 스펙.
- **프로그램 상세/참여 신청 팝업** — 프로토타입 `openJoin()`(953줄). "프로그램 선택 → 참여 팝업" 흐름의 다음 단계.
- **알림 팝업 실제 동작 / 캘린더 월별 뷰 실제 동작** — 이번엔 아이콘 배치까지 (열린 질문 C에서 최종 확정).
- **디지털 아카이브, 마이페이지 화면 콘텐츠** — 네비는 걸되 대상은 placeholder.
- **관심 계열 선택 UI** — 프로토타입은 마이페이지에서 계열을 고름(`track-picker`). 이번 화면은 이미 저장된 `career_interest`를 읽기만 함. 계열 선택 UI는 마이페이지 스펙.
- **포인트 전환/지역화폐** — 마이페이지 스펙.
- **QR 발급/인증** — 별도 스펙.

## 절대 원칙 체크
- **게임화/사행성: 영향 없음.** 랜덤박스·뽑기·학생 간 경쟁 랭킹·아바타 성장 없음. 마일스톤 스택 시각화는 "담백한 마일스톤 표시"(원칙 허용 범위). 나브 아바타는 정적 이니셜(레벨/성장 아님). 프로그램의 "인기순/popularity"는 프로그램의 인기 지표이지 학생 간 순위가 아님 — 학생 랭킹으로 변형 금지.
- **관심 계열 매칭은 개인화일 뿐 게임적 보상 아님.** OK.
- **진로 중심: OK.** 시드/추천 프로그램은 방과후·동아리·대회·기업·봉사·진로체험 등 커리어 활동만. 자습/문제풀이 없음.
- **포인트 < 포트폴리오 (원칙 4) 준수 설계:**
  - 히어로/마일스톤/개요 3종의 "디지털 포트폴리오"·성장 서사를 brand blue로 상단에 우선 배치.
  - 포인트는 (1) 나브 우측 구석 작은 amber 코인, (2) 개요 카드 3장 중 1장, (3) 카드의 `+NNN P` 뱃지에서만 amber로 절제 노출.
  - **홈 화면에 큰 포인트 잔액 배너/대시보드 만들지 말 것** (그건 마이페이지 몫).
- **포인트/지역화폐 시뮬레이션 원칙: 유지.** 이 화면은 잔액 표시(읽기)만, 전환/결제 없음.

## 인수 조건 (Acceptance Criteria)
- [ ] 로그인한 학생이 `/student`에서 상단 셸 + 히어로 + 개요 3종 + 추천 프로그램 섹션을 본다
- [ ] 캘린더 아이콘 기준일이 실제 오늘 날짜다 (하드코딩 아님)
- [ ] 나브 포인트가 로그인 학생의 실제 `points_balance`와 일치한다
- [ ] 추천 프로그램이 `is_published=true` 프로그램만 노출하고, 지난 날짜/미게시는 제외한다
- [ ] (계열 매칭 채택 시) 학생 `career_interest`와 일치하는 프로그램이 앞에 오고 "내 관심 계열" 배지가 붙는다
- [ ] 추천이 비면 빈 상태 문구가 뜬다
- [ ] 포인트 요소가 포트폴리오/성장 요소보다 시각적으로 뒤에 있다 (히어로·개요·추천이 상단, 포인트는 나브 구석·개요 1/3·카드 뱃지로만)
- [ ] 980/768/600/430에서 레이아웃이 깨지지 않고 모바일 하단 탭바가 뜬다
- [ ] 데모 시드 프로그램이 DB에 존재해 실제로 카드가 렌더된다
- [ ] 홈 화면에 큰 포인트 잔액 배너/포인트 대시보드가 없다 (원칙 4 시각 점검)
