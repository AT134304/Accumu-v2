# 프로그램 선택 화면 + 참여 신청 팝업

## 영향 범위
학생 화면 (`/student/programs` 및 홈 추천의 카드 클릭 목적지). 관리자 화면 영향 없음.
※ 단, **이 기능은 앱 최초의 "쓰기(write)" 경로**라 `participations` 테이블과 insert RLS 정책이 새로 생긴다. 권한 표면이 넓어지므로 architect-agent 검토가 반드시 선행되어야 한다 (아래 "데이터/백엔드 영향" 참고).
※ 홈(`StudentHomePage`) 추천 로직을 1곳 수정한다 (열린 질문 D) — 홈 화면 자체의 레이아웃 변경은 없음.

## 배경 / 왜 필요한가
`docs/specs/student-home.md` 확정 A/B에 따라 홈이 완성됐고, 카드/`전체 보기 →` 클릭은 **placeholder(`StudentProgramsPage.jsx`)로 라우팅만** 되고 있다. 즉 지금 앱은 "추천 8장을 보는 것"이 전부이고, 프로그램을 **찾을 수도 신청할 수도 없다.**

이번 기능은 두 가지를 채운다:
1. **프로그램 선택 화면** — 프로토타입 `screen-programs`(584~602줄). 전체 카테고리 목록·검색·필터·정렬. 홈의 추천 8장이 못 보여주는 나머지 프로그램에 도달하는 유일한 경로다.
2. **참여 신청 팝업** — 프로토타입 `openJoin()`(953줄). 상세 정보 + 신청.

특히 2번은 **v2 목표("실제로 작동하는 프로토타입")의 분기점**이다. 지금까지 앱은 읽기만 했다(`profiles_select_own`, `programs_select_published`, insert/update/delete 정책 0개). 학생이 DB에 무언가를 남기는 첫 사례가 여기서 생기고, 이후 QR·포인트·아카이브가 전부 이 `participations` 행 위에 쌓인다. 그래서 **"신청 이후" 흐름(QR 발급/인증 → 포인트 지급 → 만족도 평가)은 이번 스코프에서 잘라내되, 그 뒤에 붙을 것들이 뚫리지 않도록 권한 경계는 지금 봉인해야 한다.**

## 사용자 시나리오
1. 학생이 하단 탭바/상단 네비의 "프로그램" 또는 홈의 "전체 보기 →"를 눌러 `/student/programs`로 진입한다.
2. 상단 툴바에서 검색어("코딩", "카카오")를 입력하거나, 정렬(인기순/최신순/포인트순)을 바꾼다.
3. 필터 칩으로 활동 유형(교내 방과후·동아리·… / 교외 기업·봉사·…)이나 진로 계열(IT·SW 등)을 좁힌다.
4. "학교 내 활동" / "학교 외 활동" 그룹 아래 카테고리별 가로 스크롤 행을 넘기며 카드를 훑는다. 아래쪽 "날짜 지난 프로그램" 그룹에서 이미 끝난 활동도 확인할 수 있다.
5. 카드(또는 카드의 "참여" 버튼)를 누르면 **참여 신청 팝업**이 열린다. 상세 설명·날짜·시간·지급 예정 포인트·모집 상태 배지를 확인한다.
6. "참석 신청하기"를 누르면 신청이 DB에 저장되고, 팝업이 닫히며 토스트가 뜬다. 해당 카드의 버튼은 **"신청됨"(비활성)**으로 바뀐다.
7. 이미 신청한 프로그램을 다시 열면 버튼이 "이미 신청했습니다"(비활성)로 뜬다. 신청 불가 상태(마감/정원 초과/참석 중)나 지난 날짜 프로그램은 사유가 버튼에 표시된다.
8. 홈으로 돌아가면 신청한 프로그램은 추천 목록에서 빠져 있다 (열린 질문 D).

## 화면 구성 요소

### A. 프로그램 선택 화면 (`screen-programs`, 584~602줄)
1. **섹션 헤더** — eyebrow "Explore" / 제목 "프로그램 선택" / 서브 "카테고리를 가로로 넘기며 둘러보세요". 현재 placeholder가 이미 이 카피를 쓰고 있다.
2. **툴바** (`.toolbar` 171~183줄) — 검색 입력(`ic-search`, placeholder "프로그램, 주최, 키워드 검색") + 정렬 드롭다운(`ic-sort`, 인기순/최신순/포인트순).
3. **필터바** (`.filterbar` 186~193줄) — 칩 2줄:
   - **유형**(`ic-grid`): 전체 + CAT 8종 (칩에 `CAT[k].color` 점)
   - **계열**(`ic-target`): 전체 + TRACK 5종 (칩에 `TRACK[k].color` 점)
4. **카테고리 영역** (`#cat-area`) — 3개 그룹:
   - "학교 내 활동"(`ic-school`) → hbk/hdo/hdc/het
   - "학교 외 활동"(`ic-globe`) → ecp/evo/edc/eet
   - "날짜 지난 프로그램"(`ic-clock`, `.catgroup.past` 200~202줄) → 교내/교외 구분 없이 8종을 한 그룹에, 카테고리 라벨은 `그룹 · 이름` 형태
   - 각 카테고리 = `.catrow`(205~209줄): 헤더(색 점 + 이름 + "N개 프로그램") + **`.strip` 가로 스크롤**(210~214줄). 프로그램이 0개인 카테고리 행은 렌더하지 않는다.
5. **빈 상태** — 검색/필터 결과가 0이면 프로토타입 939줄 카피 재사용: `{조건} 검색 결과가 없습니다. 다른 키워드나 필터를 확인해보세요.` (조건 = 검색어 · 유형명 · 계열명을 ` · `로 연결)

### B. 참여 신청 팝업 (`openJoin()`, 953~975줄)
프로토타입 모달 구조 그대로:
- `.mthumb` — 카테고리 그라데이션 배경 + 카테고리 아이콘 + 닫기 버튼
- `.mbody` — `.mtag`(그룹 · 이름) / `<h3>`제목 / `.statusline`(주최 **+ 상태 배지 `badge {STATUS[].cls}`**) / `.desc`(설명) / `.infogrid`(날짜 / 시간 / 지급 예정 포인트) / 하단 CTA 버튼(`.mbtn`)
- CTA 버튼 상태 (카피는 열린 질문 F):
  | 조건 | 라벨 | 활성 |
  |---|---|---|
  | 신청 가능 | `참석 신청하기` | O |
  | 이미 신청함 | `이미 신청했습니다` | X |
  | `STATUS[status].join === false` | `{상태 라벨} — 신청할 수 없습니다` | X |
  | 지난 날짜 (`date < 오늘`) | `이미 종료된 활동입니다` | X |

### C. 토스트 (신규 공통 컴포넌트)
프로토타입 `#toast`(430~432줄) + `toast()`(1351줄) 재현. 신청 성공 시 노출, 2.6초 후 자동 사라짐. 모바일에서는 하단 탭바를 피해 `bottom:84px`(445줄).
- **[이모지 금지]** 프로토타입은 `toast('✅', ...)`로 이모지를 쓴다. CLAUDE.md 8장 위반이므로 `ic-check` duotone SVG로 대체할 것.

## 기능 요구사항 (체크리스트)

### 선택 화면 (읽기)
- [ ] `/student/programs`가 placeholder를 대체하고 공통 셸(`StudentShell`) 안에서 렌더된다
- [ ] `is_published = true` 프로그램 **전체**를 조회한다 — **홈과 달리 `date >= 오늘` 필터를 걸지 않는다** (지난 프로그램을 별도 그룹으로 보여줘야 하므로)
- [ ] 검색: 제목 + 주최 + 카테고리 표시명 대상 부분일치 (프로토타입 905줄과 동일. 설명은 검색 대상 아님). 20행 규모라 클라이언트 필터로 충분 — Supabase full-text 도입 금지
- [ ] 정렬 3종: 인기순(`popularity desc`) / 최신순(`created_at desc`) / 포인트순(`points desc`). 기본값은 열린 질문 B
- [ ] 유형 필터(CAT 8종 + 전체), 계열 필터(TRACK 5종 + 전체). 두 필터는 AND로 결합
- [ ] 교내 / 교외 / 지난 날짜 3그룹 + 카테고리별 `.strip` 가로 스크롤 행
- [ ] 프로그램이 0개인 카테고리 행·그룹은 렌더하지 않는다
- [ ] 검색/필터 결과 0건이면 조건이 포함된 빈 상태 문구 노출
- [ ] `.strip` 마우스 드래그 스크롤 동작 (`enableDrag()` 재현). 단 버튼 위에서 시작한 드래그는 무시
- [ ] 반응형 980/768/600/430 — `.strip .pcard` 폭이 430px에서 236px → 210px (453줄)

### 참여 신청 팝업 (읽기 + 쓰기)
- [ ] 카드 클릭 / 카드의 "참여" 버튼 클릭 → 팝업 오픈 (프로토타입은 둘 다 `openJoin`)
- [ ] 팝업에 `description` 표시 (현재 `programService`가 안 가져오는 필드)
- [ ] 상태 배지에 `STATUS[status].cls` 적용
- [ ] "참석 신청하기" → `participations`에 신청 행 insert
- [ ] 신청 성공 → 팝업 닫힘 + 토스트 + 해당 카드 버튼 "신청됨"(비활성)으로 즉시 반영
- [ ] 이미 신청한 프로그램은 카드/팝업 모두 "신청됨"/"이미 신청했습니다"로 비활성
- [ ] `STATUS[status].join === false`(ing/full/over)면 신청 불가 + 사유 표시
- [ ] **지난 날짜(`date < 오늘`) 프로그램은 신청 불가** (열린 질문 H — 프로토타입 버그 미재현)
- [ ] 중복 신청은 클라이언트 방어뿐 아니라 **DB 제약으로도 막힌다** (새로고침·두 탭 동시 클릭 대비)
- [ ] 신청 실패(네트워크/제약 위반) 시 조용히 넘어가지 않고 사용자에게 알린다

### 홈 연계
- [ ] 홈 추천 풀에서 이미 신청한 프로그램 제외 (열린 질문 D — ADR 0003 "향후 변경" 예고 항목)

## 데이터/백엔드 영향 (→ architect-agent 필수 검토)

**`participations` 테이블을 이번 기능에서 처음 생성한다.** CLAUDE.md 5장 필드: `id, student_id, program_id, status, entry_at, exit_at, entry_token, exit_token`.
※ 필드 목록이 CLAUDE.md에 이미 확정돼 있어 설계 자유도는 낮다. 다만 아래 3가지는 architect 결정이 필요하다.

### 이슈 1 — 권한 경계: 앱 최초의 insert 정책 (가장 중요)
지금까지의 정책은 전부 select뿐이었다. `participations`에 insert가 열리는 순간 아래를 **지금** 봉인해야 한다. 지금은 포인트 지급 로직이 없어 무해하지만, **정책을 느슨하게 열어두면 다음 QR/포인트 스펙에서 그대로 뚫린다:**

| 막아야 할 것 | 왜 |
|---|---|
| 남의 이름으로 신청 (`student_id` 위조) | `with check (student_id = auth.uid())` 없으면 임의 학생 행 생성 가능 |
| **학생이 스스로 "참여 완료" 상태로 insert** | 포인트 지급이 붙는 순간 **부정 적립 경로**가 된다. 신청 시 status는 "신청" 값으로 고정되어야 함 |
| **학생이 `entry_at`/`exit_at`/토큰 컬럼을 직접 채우기** | 같은 이유. QR 이중 인증(CLAUDE.md 6장)을 우회하는 경로 |
| 중복 신청 | 클라이언트 방어만으론 새로고침/두 탭에서 뚫린다 → DB unique 제약 권장 |
| 미게시/지난 프로그램에 신청 | `programs_select_published`는 **select만** 막는다. insert의 `program_id`는 FK만 통과하면 되므로 uuid를 알면 우회 가능(실질 리스크는 낮음 — uuid는 불투명 식별자). RLS로 막을지 architect 판단 |

- **update/delete 정책은 열지 않는다** (신청 취소는 스코프 아님 — 열린 질문 G). `mentor_students`에 정책을 0개 둔 전례와 동일 원칙: 필요할 때 연다.
- select 정책: 학생은 본인 행만(`student_id = auth.uid()`). **관리자 정책은 이번에 넣지 않는다** — QR 스캔 스펙이 올 때 별도 ADR (ADR 0003이 `programs` admin 정책을 미룬 것과 동일 근거).

### 이슈 2 — `status` 정적 필드 vs 파생 (ADR 0003 "향후 변경" 항목)
ADR 0003은 `programs.status`를 **정적 필드**로 확정했다(당시 `participations`가 없었음). 이번에 신청이 생기면 "정원이 찼는데 status는 여전히 open" 같은 모순이 이론적으로 가능해진다.

**단, 현재 시드 데이터에는 이 모순이 성립할 수 없다: `scripts/seed-programs.mjs`가 20건 전부 `capacity`를 넣지 않는다(= NULL, "정원 미정").** 정원 개념 자체가 데모에 존재하지 않으므로 정원 카운트와 status가 어긋날 여지가 없다. → 열린 질문 C 참고.

### 이슈 3 — `popularity` 첫 실사용
ADR 0003이 "사용처는 이후 프로그램 선택 화면의 인기순 정렬"로 예고한 컬럼이 여기서 처음 쓰인다. 절대 원칙 가드는 아래 "절대 원칙 체크" 참고.

### 조회/쿼리
- 데모 20행 전량 fetch 후 클라이언트 검색/필터/정렬 — ADR 0003 6번의 판단(뷰/RPC 기각) 그대로 유지.
- 신청 여부 판정을 위해 본인 `participations`를 함께 조회해야 한다. **`programs`와 조인하는 뷰를 만들지 말 것** — ADR 0003 6번이 지적한 "뷰는 기본이 정의자 권한이라 RLS 경계를 우회한다"는 함정이 여기서도 그대로 적용된다.
- 인덱스: 데모 규모상 불필요 (ADR 0003 "주의 4"와 동일). 단 unique 제약이 만드는 인덱스는 예외.

### 시드

## 절대 원칙 체크

- **게임화/사행성: 영향 있음 → 가드 필요 (인기순 정렬).**
  - **결론: 원칙 위반 아님.** `popularity`는 **프로그램의 인기 지표이지 학생 간 순위가 아니다.** ADR 0003도 이 컬럼의 사용처를 "프로그램 선택 화면의 인기순 정렬"로 미리 못박아 뒀다. 랜덤박스·뽑기·베팅·아바타 성장 없음.
  - **다만 아래를 지켜야 경쟁 서사로 오독되지 않는다 (frontend-agent 필수 준수):**
    - [ ] **`popularity` 숫자를 화면에 노출하지 않는다.** 정렬 기준으로만 쓴다. 프로토타입도 숫자를 어디에도 표시하지 않는다 — 그대로 유지.
    - [ ] **"인기 1위" / "TOP 3" / "🔥 HOT" 같은 순위·과열 라벨 금지.** 정렬 드롭다운은 "인기순" 텍스트 한 줄로 담백하게.
    - [ ] **"N명이 신청했어요" 같은 신청자 수 표시 금지.** `participations` 카운트를 화면에 내보내지 않는다.
    - [ ] **학생 단위 집계·랭킹으로 파생 금지** (ADR 0003 `popularity` 컬럼 주석 = DB에 이미 명시된 가드).
  - 계열 필터/배지는 개인화일 뿐 보상 요소가 아님. OK.
- **관리자 기능 범위: 3종 내 (영향 없음).** 학생 화면 전용 기능이며 관리자 기능을 건드리지 않는다. `participations`의 admin 정책도 이번에 열지 않는다.
- **포인트/지역화폐 시뮬레이션 원칙: 유지.**
  - 이 화면은 포인트를 **표시만** 한다 (카드 `+NNN P` 뱃지, 팝업 infogrid "참여 시 지급 포인트"). 실제 결제·계좌 연동 없음.
  - **신청해도 포인트가 지급되지 않는다** — 지급 시점은 QR 퇴장 인증(CLAUDE.md 6장 3번). 이번 스펙에 `point_transactions` insert 경로가 없다.
  - 팝업의 "· 지역화폐 전환 가능" 카피(프로토타입 971줄)는 안내 문구일 뿐 전환 동작이 아님. 유지 OK.
- **포인트 < 포트폴리오 (원칙 4): 준수.**
  - 정렬 선택지에 "포인트순"이 있으나 **기본값이 아니다** (열린 질문 B에서 기본값을 포인트순으로 두면 원칙 위반 — 선택지 중 하나로만 허용).
  - 포인트 amber는 카드 뱃지 / 팝업 infogrid 1칸에서만. **큰 포인트 배너·"포인트 많이 주는 활동" 섹션 만들지 말 것.**
  - 카드·팝업의 시각 위계는 제목/카테고리/설명(brand blue 계열)이 먼저, 포인트는 하단.
- **진로 중심: OK.** 시드 20건 전부 방과후·동아리·대회·기업·봉사·진로체험. 자습/문제풀이 없음. 필터 축(활동 유형 / 진로 계열) 둘 다 커리어 축이다.
- **QR 2회 인증(원칙 5): 훼손 없음.** 위 "스코프 아님" 참고 — 미구현이지 단순화가 아니다. 오히려 이슈 1에서 "학생이 QR을 우회해 완료 상태를 만드는 경로"를 미리 봉인한다.
- **GPS 인증: 스코프 제외 유지.** 이번 스펙에 위치 관련 요구사항 없음.

### A. 스코프 경계 — 선택 화면과 참여 팝업을 한 스펙으로 묶을까? **(가장 먼저 결정 필요)**

  - 근거 2: `participations` 최소 버전은 작다 — CLAUDE.md 5장이 필드를 이미 확정해 설계 자유도가 낮고, 정책 2개(select own / insert own)면 된다.
  - 근거 3: "프로그램을 골라서 신청한다"가 하나의 사용자 시나리오다. 여기서 끊으면 시연이 반쪽.
- **A-2: 쪼갠다.** 이번엔 선택 화면(순수 읽기, 새 테이블 없음)만 → frontend-agent 단독으로 빠르게 끝남. 참여 팝업 + `participations`는 다음 스펙.
  - 장점: 이번 스펙이 architect/backend를 안 거쳐서 빠르다.
  - 단점: 카드 클릭이 죽는다. "상세만 보여주고 신청 버튼은 다음"도 가능하지만 버튼 자리가 비어 어색하다.

### B. 정렬 3종 유지 + 기본값
- **B-1 (PM 추천): 프로토타입 그대로 — 인기순/최신순/포인트순 3종, 기본값 인기순.**
  - 홈 추천 정렬은 케빈이 "인기순 채택 안 함, 최신순"으로 확정했지만(ADR 0003 케빈 확인 1번), 그건 **앱이 자동으로 골라주는** 홈 추천 얘기였다. 여기는 **사용자가 직접 고르는 UI**라 성격이 다르다.
  - 기본값을 최신순으로 바꾸면 `popularity`의 유일한 사용처가 사실상 사라져 죽은 컬럼이 된다.
  - 포인트순은 **선택지로만** 두므로 원칙 4에 저촉되지 않는다 (기본값이면 위반).
- **B-2: 3종 유지, 기본값만 최신순** (홈과 일관).
- **B-3: 포인트순 제거**, 인기순/최신순 2종.

### C. 정원 초과 차단 + `programs.status` 정적/파생 (한 세트)
- **C-1 (PM 추천): 둘 다 이번 스코프 아님. 정원 개념을 쓰지 않고, `status`는 정적 유지.**
  - **현재 시드 20건 전부 `capacity`가 NULL이다**(`scripts/seed-programs.mjs`: "프로토타입에 정원 데이터가 없어 넣지 않는다"). 즉 정원 초과 차단을 만들어도 **데모에서 한 번도 발동하지 않는다.**
  - 신청 가능 여부의 단일 소스 = 정적 `status`(`STATUS[].join`). 정원 카운트를 아예 쓰지 않으므로 **"정원이 찼는데 status는 open" 모순이 발생할 여지 자체가 없다.**
  - `status` 파생 전환은 **"참여 확정"이 정의된 뒤**(= QR 입장 인증)가 자연스럽다. 신청만으로 정원을 세면 "신청했지만 안 온 학생"이 자리를 막는다. ADR 0003도 "participations 도입 시 **재결정**"이라 했지 "도입 즉시 전환"이라 하지 않았다.
- **C-2: 이번에 정원 도입.** `capacity` 시드를 추가하고 신청 수 기반 차단 + `status` 파생 전환.
  - 이 경우 시드 수정 + 파생 로직 + 동시성(정원 1자리에 2명 동시 신청) 처리가 따라오고, ADR 0003을 개정해야 한다. **스펙이 눈에 띄게 커진다.**

### D. 홈 추천에서 이미 신청한 프로그램 제외
- **D-1 (PM 추천): 이번에 포함.**
  - 안 하면 **신청한 프로그램이 홈에 계속 "참여" 버튼으로 뜨는 명백한 결함**이 생긴다. 신청 기능의 직접적 귀결이라 미루면 버그로 보인다.
  - `programService.fetchRecommendedPrograms` 한 곳 수정. 프로토타입 `recommended()`(837줄)의 `!isJoined && !isCompleted` 필터 그대로. ADR 0003 "향후 변경"이 이미 예고한 항목.
- **D-2: 홈은 건드리지 않는다.** 추천에 남기되 버튼만 "신청됨"으로 표시.

### E. `participations` 도입에 딸려오는 stackviz / 알림
- **E-1 (PM 추천): 둘 다 이번 스코프 아님.**
  - **stackviz(확정 G: 지금 빈 상태)는 신청만으로는 여전히 빈 상태다.** 프로토타입 `buildStack()`은 `state.completed` = **완료된 활동** 기준인데, 완료는 QR 퇴장 인증 후에 생긴다. 이번에 붙일 데이터가 애초에 없다 → **QR/포인트 스펙에서 살리는 게 맞다.**
  - **알림도 마찬가지.** 프로토타입 `confirmJoin()`이 쏘는 알림은 `notify('qr','참여 QR이 발급되었습니다', ...)`(979줄) — **QR 발급 알림**이라 QR 스펙 몫이다. 나머지 알림 시드는 "새 프로그램이 등록되었어요"로 관리자 프로그램 관리 스펙 몫. 게다가 `notifications` 테이블이 또 필요해 **한 스펙에 새 테이블 2개**가 된다.
- **E-2: 이번에 함께 연결.**

### F. QR이 없는 상태의 신청 카피 / 신청 후 동작
- **F-1 (PM 추천): 카피에서 QR 언급을 걷어내고, 신청 성공은 토스트로만 알린다.**
  | 프로토타입 | 이번 |
  |---|---|
  | `참석하기 · QR 발급받기` | `참석 신청하기` |
  | `이미 신청했습니다 · 마이페이지에서 QR 확인` | `이미 신청했습니다` |
  | 신청 → QR 발급 모달 + 알림 + 토스트 | 신청 → 토스트만 (`신청이 완료되었어요`) |
  - 근거: QR이 없는데 "QR 발급받기"라고 쓰면 **없는 기능을 약속하는 카피**가 된다.
- **F-2: QR 발급 모달까지 이번 스펙에 포함** (= A의 경계를 QR까지 넓힘).

### G. 신청 취소
- **G-1 (PM 추천): 스코프 아님.** 프로토타입에 없는 기능이고, 넣으면 `participations`에 delete/update 정책을 여는 셈이라 이슈 1의 권한 표면이 넓어진다. 시연 리셋은 `delete from public.participations;`로.
- **G-2: 넣는다.** 케빈이 **시연 중 오조작(잘못 신청)을 되돌릴 수단**이 필요하다고 판단하면 이쪽. 단 "취소 후 재신청"이 QR 토큰/포인트와 얽히므로 QR 스펙에서 다시 열어야 할 수 있다.

### H. 지난 날짜 프로그램 신청 차단
- **H-1 (PM 추천): 차단한다 (프로토타입 버그 미재현).**
  - 프로토타입 `openJoin`은 `status`만 보고 판단해서 **과거 프로그램도 `status='open'`이면 신청 버튼이 활성**이다. 시드에서 유일한 과거 행("수학 심화 탐구반")이 `status='ing'`이라 우연히 막혀 있을 뿐, 원칙적으로 구멍이다.
  - `date < 오늘`이면 신청 불가 + `이미 종료된 활동입니다`.
- **H-2: 프로토타입 그대로** (status만으로 판단).

## 인수 조건 (Acceptance Criteria)
- [ ] 학생이 `/student/programs`에서 검색·필터·정렬이 실제로 동작하는 전체 프로그램 목록을 본다 (placeholder 아님)
- [ ] 교내 / 교외 / **날짜 지난 프로그램** 3그룹이 뜨고, 지난 프로그램도 조회된다 (홈과 달리 `date >= 오늘`로 거르지 않음)
- [ ] 검색어 "카카오" 입력 시 해당 프로그램만 남는다
- [ ] 유형 칩 "교내 동아리" + 계열 칩 "IT·SW"를 함께 선택하면 두 조건을 모두 만족하는 프로그램만 남는다
- [ ] 결과가 0건이면 선택한 조건이 포함된 빈 상태 문구가 뜬다
- [ ] 정렬을 바꾸면 각 카테고리 행의 카드 순서가 실제로 바뀐다
- [ ] `.strip`이 가로 스크롤되고 마우스 드래그로도 넘어간다
- [ ] 카드를 누르면 참여 팝업이 열리고 **`description`과 상태 배지**가 보인다
- [ ] "참석 신청하기"를 누르면 `participations`에 행이 **실제로 저장된다** (새로고침해도 "신청됨" 유지)
- [ ] 같은 프로그램을 두 번 신청할 수 없다 — **DB 제약으로도 막힌다** (개발자도구로 클라이언트 방어를 우회해도 거부)
- [ ] `status`가 `full`/`over`/`ing`인 프로그램은 신청 버튼이 비활성이고 사유가 표시된다
- [ ] (H-1 채택 시) 지난 날짜 프로그램은 신청할 수 없다
- [ ] (D-1 채택 시) 신청한 프로그램이 홈 추천 목록에서 사라진다
- [ ] **신청해도 `points_balance`가 변하지 않는다** (포인트는 QR 퇴장 인증 후 — 원칙 3 시각 점검)
- [ ] **`popularity` 숫자·신청자 수·순위 라벨이 화면 어디에도 없다** (원칙 1 시각 점검)
- [ ] **큰 포인트 배너나 "포인트 많이 주는 활동" 섹션이 없다** (원칙 4 시각 점검)
- [ ] **다른 학생의 `participations`를 조회할 수 없다** (권한 점검 — 학생 B로 로그인해 학생 A의 신청이 안 보이는지)

- [ ] 980/768/600/430에서 툴바·필터바·`.strip`이 깨지지 않고, 모바일 토스트가 하단 탭바를 가리지 않는다
- [ ] 이모지가 화면에 없다 (토스트 포함 — 전부 duotone SVG)
