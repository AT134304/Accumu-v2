# 인증(로그인) 기능

## 영향 범위
공통 (학생 화면 진입점 + 관리자 화면 진입점, 로그인 이후 각 role 화면 자체는 별도 스펙)

## 선행 조건 (중요)
현재 저장소에는 코드가 전혀 없음 (Vite/React 프로젝트 미생성, Supabase 프로젝트 연결 여부 불명). 이 스펙을 architect-agent에게 넘기기 전에 **프로젝트 스캐폴딩(Vite+React 초기화, 라우터 선택, Supabase 클라이언트 설정)**이 선행되어야 함. architect-agent가 ADR로 이 초기 세팅을 먼저 기록하고 진행하는 걸 권장.

## 배경 / 왜 필요한가
Accumu는 학생/관리자 두 role이 한 앱을 함께 쓰는 구조. 기존 프로토타입은 로그인이 장식용(입력값 무관 통과)이라 실제 시연이 불가능함. 실제 Supabase Auth 기반 로그인이 들어가야 이후 모든 기능(참여 신청, QR 인증, 아카이브, 포인트)이 "내 계정 데이터"로 동작함을 보여줄 수 있음.

## 사용자 시나리오
1. 학생이 로그인 화면에서 "학생" 탭(기본 선택)에 학번·이름·비밀번호를 입력하고 로그인 → 학생 메인 화면(골격)으로 이동
2. 학생이 비밀번호를 틀리게 입력 → 에러 메시지 노출, 로그인 화면에 머무름
3. 관리자가 "관리자" 탭으로 전환 후 관리자 코드·비밀번호 입력 → 관리자 홈 화면(골격)으로 이동
4. 학생 계정 자격증명으로 "관리자" 탭에서 로그인 시도 → 거부, "선택한 유형과 일치하지 않습니다" 류 안내
5. 로그인한 상태에서 새로고침 → 세션 유지되어 재로그인 없이 동일 role 화면으로 복귀
6. 로그아웃 버튼 클릭 → 세션 종료, 로그인 화면으로 복귀 (케빈이 학생↔관리자 계정 전환하며 시연할 때 필요)
7. 비로그인 상태에서 `/student` 또는 `/admin` URL 직접 접근 → 로그인 화면으로 리다이렉트

## 기능 요구사항 (체크리스트)

### 탭 UI
- [ ] 로그인 카드 상단에 "학생 / 관리자" 세그먼트 탭 (기본 선택: 학생)
- [ ] 탭 전환 시 입력 필드셋 즉시 교체, 입력값 초기화
- [ ] 학생 탭: 학번(텍스트) · 이름(텍스트) · 비밀번호(password) 3필드
- [ ] 관리자 탭: 관리자 코드(텍스트, placeholder "예: ADM-0001") · 비밀번호(password) 2필드
- [ ] 로그인 버튼: 기존 카피 톤 유지 ("로그인 →"), 요청 중 disabled + 로딩 표시
- [ ] 에러 메시지는 폼 하단 인라인 표시, 비밀번호 필드만 초기화(학번/이름/코드는 유지)

### 인증 로직
- [ ] 학생: virtual email = `{학번}@accumu.local` 조합 후 Supabase `signInWithPassword`
- [ ] 관리자: virtual email = `{관리자코드}@accumu.local` 조합 후 동일 방식
- [ ] 로그인 성공 후 `profiles` 테이블에서 `id = auth.user.id`로 `role` 조회
- [ ] 조회된 `role`이 선택한 탭과 다르면 즉시 signOut 처리 + "선택한 유형과 계정이 일치하지 않습니다" 에러
- [ ] 인증 실패(자격증명 오류) 시 Supabase 원본 에러 메시지를 그대로 노출하지 않고 통일 문구로 치환
  - 학생 탭: "학번/이름 또는 비밀번호를 확인해주세요"
  - 관리자 탭: "관리자 코드 또는 비밀번호를 확인해주세요"
- [ ] **(확정)** 학생 로그인은 학번·이름·비밀번호 3가지가 모두 일치해야 성공. 처리 순서:
  1. `{학번}@accumu.local` + 비밀번호로 `signInWithPassword` 시도 (학번·비밀번호 검증)
  2. 성공 시 `profiles`에서 본인 행(`auth.uid() = id`) 조회
  3. `profiles.name`이 입력한 이름과 다르면 **즉시 signOut 처리** + 위와 동일한 통일 에러 문구 노출 (학번/이름/비밀번호 중 무엇이 틀렸는지 구분해서 알려주지 않음)
  4. 이름까지 일치해야 최종 로그인 성공 처리

### 라우팅 골격 (이번 스코프는 골격만)
- [ ] 로그인 성공 시 `role`에 따라 `/student` 또는 `/admin`으로 이동
- [ ] `/student` 골격 화면: "학생 메인 화면 (준비 중)" placeholder + 로그인한 학번/이름 표시 + 로그아웃 버튼
- [ ] `/admin` 골격 화면: "관리자 홈 (준비 중)" placeholder + 관리자 코드/이름 표시 + 로그아웃 버튼
- [ ] 비로그인 상태에서 `/student`, `/admin` 접근 시 로그인 화면(`/login` 또는 `/`)으로 리다이렉트
- [ ] 로그인 상태인데 자신의 role과 다른 경로 직접 접근 시(예: student가 `/admin` 입력) 자신의 role 화면 또는 로그인 화면으로 리다이렉트 (에러 노출 없이 조용히 되돌림)
- [ ] 새로고침 시 Supabase 세션이 살아있으면 자동으로 해당 role 화면 진입 (재로그인 불필요)
- [ ] 골격 화면의 나머지 요소(네비게이션 바, 포인트 표시, 알림, 캘린더 아이콘 등)는 이번 스코프 아님 — 학생 메인 화면/관리자 홈 스펙에서 별도 진행

### 로그아웃
- [ ] 골격 화면에 최소한의 로그아웃 버튼 포함 (계정 전환 시연 목적상 이번 스코프에 필수)
- [ ] 로그아웃 시 Supabase `signOut` + 로그인 화면 이동

## 데이터/백엔드 영향
- 새 테이블/필드 불필요 — CLAUDE.md에 정의된 `profiles`(id, role, code, name …) 그대로 사용
- `auth.users.id = profiles.id` 1:1 매핑 필요 → 계정 생성 시 Auth 계정과 profiles row를 함께 만드는 시드 스크립트(SQL 또는 Supabase 함수) 필요 → **architect-agent/backend-agent에게 위임**
- RLS: `profiles` 테이블에 대해 최소한 "본인 행(`auth.uid() = id`)은 select 가능" 정책 필요 (로그인 직후 role 조회를 위해) → **backend-agent에게 위임**
- `mentor_students` 매핑 데이터도 계정 시딩 시 함께 넣어둬야 함 (이번 로그인 로직 자체에는 사용 안 하지만, 이후 "담당 학생 아카이브" 기능이 이 매핑을 전제로 함) — 관리자 1명(ADM-0001)이 학생 5명 전체를 담당하는 구조로 매핑
- 데모 계정 시드 데이터(아래 "확정된 데모 계정" 표)를 SQL 마이그레이션/시드 스크립트로 고정 관리 → architect-agent가 스키마 설계 시 반영

## 스코프 아님 (Out of scope)
- 회원가입/계정 신청 UI (신규 계정은 DB에서 직접 시드, 앱 내 가입 화면 없음)
- 비밀번호 찾기/재설정 화면
- 소셜 로그인, 이메일 인증
- 관리자가 학생 계정을 앱 내에서 신규 생성/삭제하는 기능 (관리자 기능 3종에 계정관리 없음)
- 다중 관리자 계정 관리 UI
- 학생 메인 화면·관리자 홈 화면의 실제 콘텐츠 (다음 단계 별도 스펙)
- 법정대리인 동의 절차 (CLAUDE.md 11장에 따라 스코프 제외 확정 사항)

## 절대 원칙 체크
- 게임화/사행성: 영향 없음 (로그인 화면에 게이미피케이션 요소 없음, 프로토타입 카피 톤만 유지)
- 관리자 기능 범위: 해당 없음 — 로그인은 3종 관리자 기능(프로그램 관리/담당학생 아카이브/QR스캔)이 아니라 그 앞단 공통 인증 계층. 이번 스펙은 라우팅 골격까지만 다루고, 실제 관리자 기능 3종은 각각 별도 스펙에서 다룸. 확장 없음.
- 포인트/지역화폐: 해당 없음 (로그인 화면·골격 화면에 포인트 표시 없음)

## 확정된 데모 계정 (케빈 컨펌 완료)

- 계정 생성 방식: **앱 내 회원가입 UI 없음.** 모든 계정은 DB에 미리 시드해둔 데모 계정만 사용.
- 구성: **학생 5명 + 관리자 1명**, `mentor_students`는 관리자 1명이 학생 5명 전원을 담당하도록 매핑.
- 비밀번호: **전 계정 동일하게 고정** — `accumu2026`
- 학번/이름: 프로토타입(`Accumu_prototype.html` 520~521줄)에 실제로 존재하는 예시는 `10718 / 신지훈` 1건뿐이라 이를 그대로 재사용하고, 나머지 4명은 동일한 학번 체계(107xx)·흔한 3음절 국문 이름으로 새로 채움. 관리자 이름도 프로토타입에 예시가 없어 신규 지정. **이름/학번이 마음에 안 들면 backend-agent 시딩 단계 전에 언제든 교체 가능** (한 파일에서만 바꾸면 됨).

## 다음 단계 (architect-agent 인수인계 사항)

이 스펙은 확정 완료. architect-agent가 이어받아 아래 순서로 진행 권장:

1. **프로젝트 스캐폴딩** — 저장소에 코드가 전혀 없으므로 가장 먼저 필요:
   - Vite + React 프로젝트 초기화
   - 라우터 선택 및 적용 (`/login`, `/student`, `/admin` + `ProtectedRoute` 가드 구조 설계)
   - Supabase 클라이언트 설정 (env 변수 처리 방식: `.env.local` + `.gitignore` 확인)
   - 인증 상태 전역 관리 방식 결정 (React Context vs 상태관리 라이브러리) — ADR로 기록
2. **Supabase 프로젝트 연결 확인** — 기존에 만들어둔 Supabase 프로젝트(URL/anon key)가 있는지 케빈에게 확인 필요. 없으면 신규 생성부터.
3. **스키마 마이그레이션** — CLAUDE.md 5장 데이터 모델 중 이번 기능에 필요한 최소 범위(`profiles`, `mentor_students`)부터 SQL 마이그레이션 작성.
4. **RLS 정책 초안** — `profiles`: 본인 행(`auth.uid() = id`) select 허용 정책. backend-agent와 협의해 확정.
5. **계정 시딩 스크립트 설계** — 위 표의 6개 계정을 Supabase Auth + `profiles` + `mentor_students`에 함께 넣는 방법 결정:
   - Supabase Admin API(service role key)로 서버사이드 스크립트 작성 vs Dashboard 수동 생성 + SQL insert 병행
   - 시딩 스크립트 저장 위치 제안: `supabase/seed.sql` 또는 `scripts/seed-accounts.ts` (architect-agent가 최종 결정, ADR에 기록)
6. 위 설계가 끝나면 **backend-agent**로 넘겨 실제 마이그레이션 SQL·RLS·계정 시딩 실행, 이어서 **frontend-agent**로 로그인 화면·라우팅 골격 구현.

## 인수 조건 (Acceptance Criteria)
- [ ] 로그인 화면에 학생/관리자 탭이 있고 탭 전환 시 필드가 정확히 바뀐다
- [ ] 유효한 학생 자격증명으로 로그인 시 `/student` 골격 화면으로 이동한다
- [ ] 유효한 관리자 자격증명으로 로그인 시 `/admin` 골격 화면으로 이동한다
- [ ] 잘못된 자격증명 입력 시 에러 메시지가 뜨고 로그인 화면에 머무른다
- [ ] 학생 계정으로 관리자 탭 로그인 시도(또는 반대) 시 거부되고 안내 메시지가 뜬다
- [ ] 학번·비밀번호가 맞아도 이름이 틀리면 로그인이 실패한다 (예: 10718 / 잘못된이름 / 정확한 비밀번호 → 실패)
- [ ] 로그인 상태에서 새로고침해도 세션이 유지되어 자동으로 같은 role 화면에 재진입한다
- [ ] 로그아웃 버튼 클릭 시 세션이 종료되고 로그인 화면으로 돌아간다
- [ ] 비로그인 상태에서 `/student`, `/admin` 직접 접근 시 로그인 화면으로 리다이렉트된다
- [ ] 확정된 데모 계정 6개(학생 10718~10722, 관리자 ADM-0001, 비밀번호 전부 `accumu2026`)가 DB에 시드되어 실제로 로그인 가능하다
